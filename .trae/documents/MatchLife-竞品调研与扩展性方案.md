# MatchLife 竞品调研与扩展性方案（Draft）

## 1. 竞品/同类系统常见能力（最佳实践提炼）
面向“赛事管理 + 实时比分 + 统计”的系统，行业里普遍会收敛到以下能力组合：

- **实时比分与赛程**：支持比赛进行中的实时记分/更新、赛程列表、场地/时间维度筛选、自动推进与更新榜单/对阵树。
- **站位与晋级结构**：支持循环赛/小组赛/淘汰赛/双败等赛制的展示与状态推进。
- **排名与积分体系**：按赛事规则生成排名/积分；支持不同口径（赛段、组别、地区、时间窗口）。
- **选手主页（Player Profile）**：沉淀选手档案、历史参赛记录、关键指标（胜率、对阵记录、近期状态）。
- **对阵与头对头（H2H）**：常见的“对手分析/历史交锋”能力。
- **快速搜索与过滤**：全局搜索框 + 标签筛选/条件筛选是高频入口。
- **数据导出与共享**：CSV 导出、分享链接、公众号/社群传播入口。

参考：多家赛事管理与实时记分产品都强调“实时记分、自动更新榜单/对阵结构、选手主页与统计沉淀”等组合能力。[1][2][5]

## 2. MatchLife 的产品定位调整（建议）
当前阶段 MatchLife 更适合定位为：

- **“数据聚合/检索/统计工具（Public Read）”**：以免登录查询为核心，解决“快速查人/查赛/查成绩”。
- **“轻赛事中台雏形（Data Hub）”**：先把数据采集/标准化/入库打通，再逐步叠加“赛制结构、积分排名、选手画像”等能力。

## 3. 信息架构（IA）与页面重构建议
### 3.1 MVP（你当前要做的第一阶段）
- **首页（搜索）**：一个搜索框 + 快捷过滤 + 最近检索历史 + 结果列表。
- **比赛详情**：对阵双方、比分/局分、场地、时间、组别、阶段、来源。
- **赛事看板**：核心 KPI + 组别/赛事/场地分布 + 近期更新趋势。
- **数据同步**：同步任务列表、每次同步的 pulled/upserted/invalid/inserted/updated/skipped。

### 3.2 扩展阶段（第二阶段开始做“多赛事、多来源”）
- **赛事中心（Tournaments）**：赛事列表、来源、状态（进行中/已结束）、数据覆盖范围。
- **选手中心（Players）**：可按姓名/别名/机构检索；选手主页展示生涯。
- **对阵分析（H2H）**：A vs B 的历史交锋、胜负趋势、常见比分。
- **积分/排名（Standings）**：按赛事规则生成排行榜；支持导出。

## 4. 统计表与口径（建议改造）
### 4.1 赛事看板 KPI（建议固定）
- 比赛总场次、完赛场次、进行中场次
- 参赛选手数（去重）、赛事数
- 今日新增比赛/今日更新比赛
- 数据新鲜度（最近一次 source_updated_at）

### 4.2 分布图（建议）
- 组别分布 TopN（U12/U13/U14…）
- 场地分布（场地 1..N）
- 赛段分布（小组/淘汰/排位等，以源字段映射）
- 近期更新趋势（按小时/天聚合）

### 4.3 选手统计（建议）
- 总参赛场次、胜场、负场、胜率
- 近 30 天表现（胜率/胜负）
- 常见对手 TopN、常见比分 TopN

## 5. 数据模型扩展（从“单表 matches”走向可扩展）
### 5.1 建议新增核心表
- `tournaments`：赛事维度（name, source, source_game_id, start/end, city/location, status）
- `players`：选手维度（name, aliases, gender, birth_year, org/club）
- `matches`：仍为事实表，但增加 `tournament_id`、`player_ids`（或关联表）
- `sync_sources`：数据源配置（ymq/人工导入/其它平台），支持“手动触发 + 定时抓取 + 频率策略”

### 5.2 多来源采集策略
- **自动采集**：按数据源定义轮询策略（进行中高频、已结束低频）。
- **手动采集**：在“赛事中心”录入一个新的来源链接/比赛ID，一键抓取并纳入库。
- **人工导入**：提供 CSV 模板导入，适配业余赛事/线下赛。

## 6. 实施建议（最小成本）
- 先实现“赛事中心”最小表：只存 `source + source_game_id + name`，用于支持后续多赛事抓取。
- 把“同步”从单赛事固定参数升级为：`sync_sources` 驱动（每条 source 一套参数与频率）。
- 统计优先 SQL 聚合；复杂统计再引入物化视图/预计算。
