# MatchLife 部署路径基线

## 目标

明确 MatchLife 在不同环境下允许使用的子路径，并说明本地构建、UAT 发布、生产发布之间的对应关系，避免再次出现错误基路径构建被发布到线上目录的问题。

## 当前路径矩阵

- UAT 域名：`tools.cheersai.cloud`
- UAT 页面入口：`/7smile-matchlife/`
- UAT 静态资源：`/7smile-matchlife/assets/*`
- UAT 微信入口：`/7smile-matchlife/api/wechat/*`
- UAT 数据代理：`/7smile-matchlife/supabase/*`

- 生产域名：`7smile.dlithink.com`
- 生产页面入口：`/match-life/`
- 生产静态资源：`/match-life/assets/*`
- 生产微信入口：`/match-life/api/wechat/*`
- 生产数据代理：`/match-life/supabase/*`

## 单点配置来源

- UAT shell 权威入口统一为 `TECH_SCRIPT_DIR=/Users/FYP/Documents/WorkSpace/CheersAI/CheersAI - docs/技术/scripts/matchlife-uat`。
- 仓库内 `scripts/matchlife-uat/*.sh` 仅保留兼容转发，不再作为路径基线文档的默认入口。

- 前端构建基路径：`vite.config.ts` 中的 `APP_BASE_PATH`
- UAT 构建入口：`$TECH_SCRIPT_DIR/01-build.sh`
- UAT 推送入口：`$TECH_SCRIPT_DIR/02-push.sh`
- UAT 远端发布入口：`$TECH_SCRIPT_DIR/03-release.sh`
- 远端落地脚本：`$TECH_SCRIPT_DIR/deploy-matchlife.sh`

## 发布约束

- 构建时必须显式传入环境对应的 `APP_BASE_PATH`
- UAT 发布默认值固定为 `/7smile-matchlife/`
- 生产发布默认值应显式改为 `/match-life/`
- 不允许把 `/match-life/assets/*` 的构建产物发布到 UAT 的 `/7smile-matchlife/` 目录
- 不允许把 `/7smile-matchlife/assets/*` 的构建产物发布到生产的 `/match-life/` 目录

## 已固化保护

- `$TECH_SCRIPT_DIR/02-push.sh`
  - 上传前检查 `dist/index.html` 中所有根路径资源是否都落在 `${APP_BASE_PATH}assets/` 下
- `$TECH_SCRIPT_DIR/deploy-matchlife.sh`
  - 远端覆盖 `current/` 后再次检查 `index.html` 中资源前缀是否与 `${APP_PATH}/assets/` 一致

## 排查顺序

当监控提示 `asset path escaped app base` 时，按以下顺序检查：

1. 本地 `dist/index.html` 是否引用了错误前缀
2. `APP_BASE_PATH` 是否与当前环境一致
3. 远端 `current/index.html` 是否仍残留旧版本内容
4. `04-monitor.sh` 的 `APP_PATH` 是否与发布目标一致

## 当前结论

- `/7smile-matchlife/` 与 `/match-life/` 并非天然冲突，而是分别服务于 UAT 与生产
- 真正的问题是“错误环境的构建产物被发到了当前环境的站点目录”
- 后续应以“路径与环境一一对应”为原则，而不是简单删除另一套路径
