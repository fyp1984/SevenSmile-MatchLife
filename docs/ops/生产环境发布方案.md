## 发布约束

- 生产构建必须显式使用 `APP_BASE_PATH=/match-life/`。
- 不允许将 `/7smile-matchlife/assets/*` 的构建产物发布到生产目录。
- 生产与 UAT 必须围绕同一个 `branch + commit SHA` 做基线锁定。
- 生产发布前不得直接修改服务器上的前端产物或运行时脚本；所有修复都先落在本地正式工程。

## 构建与数据链路

- 生产构建阶段统一复用 `TECH_SCRIPT_DIR=/Users/FYP/Documents/WorkSpace/CheersAI/CheersAI - docs/技术/scripts/matchlife-uat` 下的 `01-build.sh`，以保持与 UAT 相同的 shell 实现。

生产构建命令：

```bash
APP_BASE_PATH=/match-life/ bash "$TECH_SCRIPT_DIR/01-build.sh"
```

生产环境的数据回源原则：

- `SUPABASE_REST_UPSTREAM=http://127.0.0.1:8000`
- `SUPABASE_URL=http://127.0.0.1:8000`
- `SUPABASE_REST_HOST_HEADER=127.0.0.1`

这样可保持页面对外 URL 不变，同时避免生产链路回源公网地址。

## 推荐发布顺序

### 1. 基线锁定

发布前先确认：

1. 当前本地分支正确
2. 工作区干净或已知改动明确
3. 已完成 `git fetch --prune`
4. 如本地领先远端，先完成推送
5. 记录本次发布的 branch 与 commit SHA

### 2. 基础设施就绪

建议先完成证书、Nginx location 和运行目录准备，再进行完整发布。若基础设施尚未完全补齐，至少先验证：

- `/etc/nginx/ssl/7smile.dlithink.com_bundle.pem`
- `/etc/nginx/ssl/7smile.dlithink.com.key`
- `include /etc/nginx/locations/*.conf`
- MatchLife 生产 location 已纳入网关

### 3. 完整发布

生产发布应沿用与 UAT 同风格的阶段化流程：

1. `check-env`
2. `build`
3. `push`
4. `release`
5. `monitor`

如果后续正式补齐 `scripts/matchlife-prod/`，应保证职责与 UAT 脚本一一对应。

## 验收口径

发布完成后至少验证：

- `https://7smile.dlithink.com/match-life/` 返回 200
- `/match-life/stats` 刷新不 404
- `/match-life/assets/*` 可访问
- `/match-life/api/wechat/*` 不出现 502/504
- `7smile-matchlife-wechat-access.service` 为 `active`
- `7smile-matchlife-sync.service` 为 `active`
- `heartbeat.json` 持续刷新

## 回滚策略

回滚优先级如下：

1. 将 Nginx 流量或入口切回上一版
2. 将 `current` 符号链接回退到上一版 release
3. 重启 `wechat-access` 与 `sync` 服务
4. 重新执行巡检

建议始终保留至少两个历史版本目录，确保可以快速回退。
