# 2026-05-20 UAT 路径与日志修复说明

## 背景

本轮处理了两类问题：

1. MatchLife UAT 页面仍在引用 `/match-life/assets/*`，与当前 UAT 子路径 `/7smile-matchlife/` 不一致
2. MatchLife watcher 与 Desktop worker 的高频日志持续冲击 UAT `syslog`

## 路径问题根因

- 本地源码当前构建默认已是 `/7smile-matchlife/`
- 但远端 `current/index.html` 一度仍带有 `/match-life/assets/*`
- 说明问题不在前端源码本身，而在发布链没有对构建产物基路径做强校验，导致旧或错误环境产物存在被继续上线的风险

## 本次修复

- 当前文档中的 UAT shell 权威入口统一为 `TECH_SCRIPT_DIR=/Users/FYP/Documents/WorkSpace/CheersAI/CheersAI - docs/技术/scripts/matchlife-uat`；仓库内同名 `.sh` 仅保留兼容层。

### 1. 发布前校验

在 `$TECH_SCRIPT_DIR/02-push.sh` 中新增 `validate_dist_base_path()`：

- 读取 `dist/index.html`
- 提取所有根路径 `src/href`
- 校验所有静态资源都落在 `${APP_BASE_PATH}assets/` 下
- 若存在 `/match-life/assets/*` 之类错误前缀，直接阻断推送

### 2. 远端落地后校验

在 `$TECH_SCRIPT_DIR/deploy-matchlife.sh` 中新增 `validate_index_base_path()`：

- 在远端 `current/index.html` 覆盖完成后再次读取 HTML
- 校验资源前缀是否与当前 `APP_PATH` 一致
- 若部署目录里的 HTML 仍指向错误路径，直接中止发布

### 3. watcher 日志节流

在 `scripts/watch-ymq.mjs` 中引入：

- 摘要日志输出
- 无增量时最小打印间隔 `MATCHLIFE_LOG_MIN_INTERVAL_MS`
- 状态变化与实际新增/更新优先打印

目的：

- 避免将完整大对象或高频空增量心跳持续写入 `syslog`

## 验证口径

- 本地 `APP_BASE_PATH=/7smile-matchlife/ bash "$TECH_SCRIPT_DIR/01-build.sh"`
- 确认 `dist/index.html` 仅包含 `/7smile-matchlife/assets/*`
- 再执行：
  - `bash "$TECH_SCRIPT_DIR/02-push.sh"`
  - `bash "$TECH_SCRIPT_DIR/03-release.sh"`
  - `bash "$TECH_SCRIPT_DIR/04-monitor.sh"`

## 备注

- 生产环境仍可能合法使用 `/match-life/`
- 因此修复策略不是“删除 `/match-life/`”，而是“禁止当前环境发布错误基路径产物”
